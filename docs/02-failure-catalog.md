# 02 — Failure Catalog

> The authoritative catalog of real failure modes across the 6 Datto/Kaseya products — the **source of truth** for remediation actions, content, and the automation engine.
> Part of the Kaseya Resolution Center spec set — see [INDEX](INDEX.md).

## 0. How to read this doc

This catalog is **synthesized faithfully from research** — see [`research/failure-catalog.json`](research/failure-catalog.json) (135 structured failure modes) and [`research/00-failure-catalog-digest.md`](research/00-failure-catalog-digest.md). Each failure mode is a real, evidence-backed reliability problem drawn from Datto KB articles, MSP forums (community.datto.com, r/msp, r/Datto), G2/Capterra/TrustRadius reviews, and documented incidents.

This is the **upstream document** for the rest of the spec set:

- The **distinct remediation actions** in [§8](#8-consolidated-cross-product-remediation-actions) are the seed for the action library that the automation engine executes, chains, scopes, and saves as playbooks — see [domain model](05-domain-model.md) and [automation engine](06-automation-engine.md).
- The **failure categories** in [§9](#9-failure-category-taxonomy) are the canonical filter/group dimension used in the triage tables, the issue queue, and `nuqs` URL state — see [information architecture](04-information-architecture.md).
- The **frequency/severity** fields drive the worst-real-child fleet rollup and severity sort (Failed > Warning > Offline > Syncing > Paused > Protected) defined in the design system.

### 0.1 Field definitions (every failure mode carries these)

| Field | Meaning |
|---|---|
| **ID** | Stable slug used as the key in code and content (`storage-pool-full-backups-skipped`). |
| **Category** | One of the canonical categories in [§9](#9-failure-category-taxonomy). The filter/group dimension. |
| **Symptoms** | What the technician *sees* — alerts, error banners, observable behavior. Used for matching/diagnosis. |
| **Root causes** | The underlying *why*. Multiple per mode; the engine narrows these via diagnostic actions. |
| **Frequency** | `very common` › `common` › `occasional` › `rare`. Drives default sort + "what to fix first." |
| **Severity** | `critical` › `high` › `medium` › `low`. Maps to status tokens (critical/high → Failed, medium → Warning, low → cosmetic/info). |
| **Current resolution** | How it's solved today, **including existing Datto-side automation** (e.g. auto diff-merge after 5 failed screenshots, DBD crash-consistent fallback, auto retry/re-pair). This is what we improve on. |
| **Self-serve steps** | Steps a technician can complete *without* Datto Support — the basis for one-click/guided actions. |
| **Human-in-loop** | The judgement/physical/business decision that **cannot** be automated. The gate-for-approval boundary. |
| **Automatable?** | `true` = at least partially one-click/auto-remediable. `false` = primarily human/Support-led; we assist (pre-fill, collect logs, checklist) but cannot fully automate. |
| **Candidate actions** | The discrete remediation ACTIONS this mode implies. Feed [§8](#8-consolidated-cross-product-remediation-actions). |
| **Error strings** | Real alert/error codes for matching & display in `mono`. (See [§10](#10-error-string-index).) |

### 0.2 Severity ↔ status token mapping

Failure severity is **not** the same as live asset status, but the catalog drives how an open failure is rendered:

| Catalog severity | Status token shown | Notes |
|---|---|---|
| `critical` | `Failed` (#DC3545) | Data-loss risk, DR-blocking, or fleet-wide outage. Always top of severity sort. |
| `high` | `Failed` (#DC3545) | Backups stopped / recovery blocked for this asset. |
| `medium` | `Warning` (#FFC107) | Degraded (e.g. crash-consistent fallback, slow merge) but protection continues. |
| `low` | `Warning` / info | Often cosmetic (screenshot timing false-failures). Surface but de-emphasize. |

> **Status is never color-only.** Every status renders dot + icon + label per the design system.

## 1. Product 1 — Datto BCDR (SIRIS / ALTO appliances)

On-prem image-based backup + DR appliance line. Image backups (agent + agentless VMware/Hyper-V) → local ZFS pool → off-site replication to Datto Cloud. Recovery via screenshot verification (boot tests), local/cloud virtualization, BMR, file/folder restore. Uses Inverse Chain Technology + ransomware detection. **Strong core engine** (G2 ~4.6, TrustRadius ~9.2); friction is **operational, not data-loss** — storage fills, VSS failures, cosmetic screenshot failures, off-site lag, agent pairing breakage. **22 failure modes.**

> Datto-side automation already present: **auto diff-merge after 5 consecutive failed screenshots** (IRIS 4.0+, if a success occurred within the last month), **DBD crash-consistent fallback** when VSS export fails, **auto-retry/re-pair** on `AGT0900`.

## 7. Cross-product observations

A few patterns from the 135-mode synthesis that shape the engine and IA:

1. **Two product families, two action vocabularies.** The **backup-engine products** (BCDR, DEB v1, DEB v2, Cloud) share an *infrastructure* action vocabulary (VSS, agent comms, diff-merge, screenshot, off-site sync, BMR, virtualization). The **SaaS-backup products** (SaaS Protect, Spanning) share a *cloud-API* vocabulary (OAuth/consent, seat/license lifecycle, API throttling, re-sync/re-index, restore-into-source). The action library must namespace these but allow shared primitives (Open/assemble Support escalation, Force re-run, Schedule off-peak).
2. **"Failure" ≠ data loss.** The single largest cluster — **screenshot/local verification** + much of **API throttling** — is *cosmetic or transient*. The app must let techs **classify cosmetic vs real** and suppress known-limitation noise, or it amplifies alert fatigue.
3. **Diff-merge is the universal self-heal** for the backup-engine products. It appears as a candidate action in BCDR, DEB v2, and Cloud across VSS, screenshot, integrity, and virt failures. Surfacing it as a first-class, progress-tracked action with a **tunable auto-trigger threshold** (default: after 5 consecutive failed screenshots) is high-leverage.
4. **OAuth/consent + seat lifecycle is the universal self-heal** for the SaaS products — re-authorize, Protect All, force re-discovery, unseat-but-retain. The **EWS→Graph May 30, 2026** deadline ([5.2](#5-product-5--datto-saas-protection-datto-saas-protect)) is the single highest-stakes deadline-driven action in the catalog and deserves a dedicated bulk-reauth queue with a countdown.
5. **Support escalation is a "human-in-loop" action, not a dead end.** ~20 modes are `auto: false` and converge on Support. The value-add is **assembling the escalation** (logs, version, error, steps tried) and **tracking SLA** — turning an unavoidable human step into a one-click pre-filled package.
6. **Encryption passphrases and irreversible operations** ([1.17](#1-product-1--datto-bcdr-siris--alto-appliances), [4.9](#4-product-4--datto-cloud-continuity--dr), [4.15](#4-product-4--datto-cloud-continuity--dr), [4.21](#4-product-4--datto-cloud-continuity--dr), [6.6](#6-product-6--spanning-backup-kaseya)) are the hard wall for automation: they **always** need a human and an explicit confirmation gate. These map directly to the approval-gating mandate in the [automation engine](06-automation-engine.md).

## 11. Open decisions / questions flagged

1. **`Diff-Merge / Chain Rebuild` as its own category vs a Backup Chain sub-filter** ([§9.2](#92-normalization-notes)). It holds only 1 raw mode but names a cross-product *action* and *operational state*. Recommendation: keep as a distinct **state badge** but allow it to roll up under Backup Chain in the category filter. Needs sign-off from IA.
2. **Status-token mapping for `low`-severity cosmetic failures** ([§0.2](#02-severity--status-token-mapping)). Screenshot timing false-failures are technically "Failed" in Datto but should not paint an asset red in our rollup. Proposal: a derived **"cosmetic/known-limitation"** sub-state that renders as info/Warning, not Failed. Confirm with design.
3. **Gating defaults in [§8](#8-consolidated-cross-product-remediation-actions)** are a first-pass proposal (⛔/⚙️/🔎). The authoritative approval policy lives in the [automation engine](06-automation-engine.md); this catalog should be treated as the *input*, not the final word, on which actions auto-run vs require approval.
4. **EWS→Graph May 30, 2026 deadline** ([5.2](#5-product-5--datto-saas-protection-datto-saas-protect)) is a real, dated event. For a mock with `currentDate` 2026-06-22 the deadline has **passed** — mock data should model both pre- and post-deadline states (some orgs reauthorized, some Exchange backups now stopped) to make the bulk-reauth queue demo realistic.
