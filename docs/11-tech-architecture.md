# Tech Architecture

The front-end architecture for the Kaseya Resolution Center mock: stack rationale, folder structure, state model, mock services, and the lint/CI machinery that enforces the design mandates. Part of the Kaseya Resolution Center spec set — see [INDEX](INDEX.md).

> Scope reminder: this is a **front-end MOCK** — no backend, no auth, no network. Everything is generated and served in-memory/localStorage. Architecture optimizes for **fidelity + iteration speed + zero infra**, not server scalability. The mock data layer is the *seam* a real API would later replace.

Cross-links: [information architecture](04-information-architecture.md) · [domain model](05-domain-model.md) · [data model & mock data](06-data-model-and-mock-data.md) · [automation engine](07-troubleshooting-and-automation-engine.md) · [design system](03-design-system.md) · [Storybook design system](storybook-design-system.md) · [component inventory](10-component-inventory.md).

## 1. Stack — confirmation & rationale

The stack is **locked**. The table below records *why* each piece earns its place in a mock troubleshooting console.

| Layer | Choice | Rationale (mock-specific) |
|---|---|---|
| Framework | **Next.js 15 App Router + TypeScript** | File-based routing *is* our [IA skeleton](04-information-architecture.md); nested layouts give a persistent app shell (white-nav sidebar + topbar + action cart) that never remounts on navigation; `loading.tsx` yields free skeletons during simulated latency. TS is non-negotiable — the mock's whole value is a once-typed [domain model](05-domain-model.md) reused everywhere. Run client-heavy; App Router is structure, not a data-fetching story. |
| Styling | **Tailwind CSS v4** | CSS-first `@theme` config + OKLCH default → perceptually-uniform dark mode (our slate) "just works." Tokens are declared once in `globals.css` and exposed as utilities. Pairs natively with shadcn. |
| Components | **shadcn/ui** (Radix primitives, copy-in source) | We own the source, so we re-skin to the Datto look (Curious Blue primary, hairline borders, 8px radius) without fighting a library. Local files = edit, not theme-around. |
| Icons | **lucide-react** | shadcn default; tree-shakeable; covers shield / server / hard-drive / cloud / alert metaphors for the 6 products. |
| Charts | **Recharts v3** (+ thin **Tremor**-style KPI tiles) | shadcn's `chart` block wraps Recharts and reads `--chart-1..5` tokens, so charts inherit theme automatically. Tremor only for fast KPI tiles on the dashboard — never the primary surface (tables are). |
| Data grid | **TanStack Table v8** (headless) + shadcn `Table` markup | The dense table is the core surface ("last 10 backups" dot-strip, sticky header/col, bulk toolbar, faceted filters, column visibility). Headless = it manages state, renders nothing; shadcn supplies markup. Anything less guarantees a rewrite. |
| List virtualization | **@tanstack/react-virtual** | Fleets of hundreds of assets/jobs/alerts/runs — virtualize so the mock stays buttery (see [§8](#8-performance--large-mock-fleets)). |
| Client state | **Zustand** (+ `persist` middleware) | The action cart / chains / playbooks / policies are cross-route, frequently mutated (drag-reorder, param edits, scope toggles), and must survive reloads. `persist` hydrates synchronously from localStorage before render. Context would re-render the subtree on every keystroke. |
| URL state | **nuqs** | Type-safe, `useState`-like, query-string-backed (~6kB). Makes table filters, active tab, search, and the open detail-drawer **deep-linkable & shareable** — the single biggest "this feels real" detail. Powers [saved views](04-information-architecture.md). |
| Forms | **react-hook-form + zod** | Playbook editor, action-param entry, approval forms. zod schemas double as **fixture validators** — one schema guards both the form and the generated data. shadcn `Form` is RHF-native. |
| Toasts | **Sonner** | "Action queued", "Playbook saved", "Remediation dispatched (mock)", "Auto-remediation fired" feedback. |
| Dates | **date-fns** | Deterministic, tree-shakeable; relative-time ("2h ago") is core to alert/run/backup feeds. |
| Design-system docs/test | **Storybook 10** (`@storybook/nextjs-vite`) | See [Storybook design system](storybook-design-system.md). Atomic structure, 100% story coverage, a11y gate, Vitest browser-mode tests, themes addon. Imports the *same* `globals.css` so stories render pixel-identical to the app. |

### Determinism caveat (call it out — it shapes the mock layer)
Next may prerender at build time. **`Math.random()` and `Date.now()` are forbidden in any module that can run at build/SSR** (hydration mismatch + non-reproducible fixtures). The rule:

- **Build/SSR-safe fixtures** → generated by a **seeded PRNG** (same seed → byte-identical data forever; stable screenshots & Storybook snapshots).
- **Runtime-only effects** (a live-feeling alert ticking in, a run streaming output) → randomness allowed **only inside `useEffect` / event handlers**, never during render.

## 3. State architecture

Two distinct planes. Keep them separate — that separation is what lets the mock later grow a real backend.

```
┌─────────────────────────────── STATE PLANES ───────────────────────────────┐
│                                                                             │
│  SERVER-ish DATA            URL STATE (nuqs)          CLIENT STATE (Zustand)│
│  (mock/query.ts)            view/filter/selection      cross-route + persist│
│  ─────────────────          ────────────────────       ─────────────────────│
│  read-only fixtures         status filter, search,     action cart + chain  │
│  paginated/filtered         active tab, open drawer    saved playbooks       │
│  in-memory "DB"             id, sort, page, density     auto-remediation pol.│
│  (would become fetch)       → deep-linkable/shareable  saved views, approvals│
│                                                         → persist→localStorage│
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3a. Server-ish mock data layer (read path)
`mock/query.ts` is the only place pages read entities from. It mimics a paginated/filterable backend and is the API seam. Pages call e.g. `getAssets(q)` in a Server-ish thin wrapper or directly in a client component; an optional `await delay(...)` exercises the `loading.tsx` skeletons. **No client store ever holds the fixture rows** — fixtures stay in the data layer so they can't drift or duplicate.

### 3b. URL state (nuqs) — the "view" plane
Everything that defines *what the user is looking at* lives in the query string so any view is shareable:

| Key (nuqs parser) | Holds | Routes |
|---|---|---|
| `q` (`parseAsString`) | search query | all tables |
| `status` (`parseAsArrayOf`) | faceted status filter | fleet, backups, alerts |
| `product` (`parseAsArrayOf`) | which of the 6 products | fleet, backups |
| `severity` (`parseAsArrayOf`) | alert severity facet | alerts |
| `sort` (`parseAsString`) | `col:dir` | all tables |
| `page` / `pageSize` (`parseAsInteger`) | pagination | all tables |
| `tab` (`parseAsStringEnum`) | active detail tab | asset/job detail |
| `open` (`parseAsString`) | open drawer/sheet id (asset or alert) | fleet, alerts |
| `density` (`parseAsStringEnum`) | comfortable/compact | all tables |

A **saved view** = a named snapshot of these params (stored in the `saved-views` Zustand store); applying it pushes the params back onto the URL. See [IA → saved views](04-information-architecture.md).

### 3c. Client stores (Zustand) — the "intent" plane
Frequently-mutated, cross-route, persisted. Each store uses `persist` with `partialize` (keep transient flags out of storage) and `version` + `migrate` (evolve shape without bricking a demoer's saved state). All gate hydration-sensitive UI behind a `useHasHydrated()` flag (subscribe to `persist.onFinishHydration`) so the persisted cart-count badge never causes a hydration mismatch.

```ts
// stores/action-cart.ts  (centerpiece — see automation engine doc)
type Scope = 'once-this' | 'once-matching' | 'always';   // apply once to this asset /
                                                          // once to all matching / always (auto-remediation)
interface ChainStep {
  uid: string;                 // instance id (an action may appear twice)
  actionId: string;           // → action catalog
  params: Record<string, unknown>;
  scope: Scope;               // per-step override of cart default
  requiresApproval: boolean;  // gating flag (mock approval queue)
}
interface ActionCartState {
  targets: string[];                       // selected asset ids (the scope target set)
  steps: ChainStep[];                      // ordered remediation chain
  defaultScope: Scope;
  addAction:   (actionId: string) => void;
  removeStep:  (uid: string) => void;
  reorder:     (from: number, to: number) => void;
  setStepScope:(uid: string, s: Scope) => void;
  setTargets:  (ids: string[]) => void;
  clear:       () => void;
  dispatch:    () => Promise<Run[]>;       // → mock/runner.ts
}
```

| Store | localStorage key | Holds | Persisted slice (`partialize`) |
|---|---|---|---|
| `action-cart` | `dcc.action-cart.v1` | targets, chain steps, default scope | all except `dispatch` in-flight flags |
| `playbooks` | `dcc.playbooks.v1` | seeded + user playbooks | full list |
| `automation-policies` | `dcc.policies.v1` | "always" standing rules (auto-remediation) | full list |
| `saved-views` | `dcc.saved-views.v1` | named filter/sort snapshots per route | full list |
| `approvals` | `dcc.approvals.v1` | pending/decided approval requests | full list |
| `ui` | `dcc.ui.v1` | sidebar collapsed, density default, theme | full |

**Closing the loop:** "Save as playbook" reads `action-cart` state → writes to `playbooks`. "Run playbook" loads steps **back into** `action-cart`. An `always`-scoped chain at dispatch writes a rule into `automation-policies`; a runtime ticker (see [§4c](#4c-fake-realtime-updates)) can fabricate a matching alert to demo the policy auto-firing. Approval-gated steps land in `approvals` instead of dispatching immediately. Details in [automation engine](07-troubleshooting-and-automation-engine.md).

**Settings → "Reset demo state"** clears all `dcc.*` keys and reseeds, so the mock is always presentable for the next demo.

## 4. Mock services

Three cooperating pieces: a deterministic generator, a simulated runner, and a fake realtime ticker.

### 4a. Deterministic seeded generator
No `Math.random` at module scope. A tiny dependency-free hash→PRNG (`mock/prng.ts`), with one **named RNG stream per entity** so streams are independent and stable:

```ts
export const SEED = 'datto-care-center-2026';                 // mock/seed.ts
export function rng(namespace: string) {                       // mock/prng.ts
  return mulberry32(xmur3(`${SEED}:${namespace}`)());         // xmur3 + mulberry32: pure bit ops, build-safe
}
```

Generators draw from curated value pools so output reads like real Datto data, not lorem ipsum — appliance names `SIRIS-NYC-01`, agents `LON-WS-0142`, orgs `Northwind Traders` / `Contoso Health`, error codes in mono. **Data is correlated, not just random** (this is what sells realism):

- A `Failed` asset carries `openAlerts > 0`, a recent failed backup, and a lower 30-day success rate.
- `Offline` appliances → `lastSeen` days ago and their backup jobs read `missed`.
- A fleet/org rollup status = **worst real child state** (severity order: Failed > Warning > Offline > Syncing > Paused > Protected), matching the [design system](03-design-system.md).
- Alerts reference real `assetId`s; runs reference real `actionId`+`assetId`; playbooks chain real actions — referential integrity holds because cross-links draw from the same seeded streams.

Counts target a dense-but-instant console: ~480 assets / ~620 backup jobs / ~900 alerts / ~1,500 runs / ~40 actions. Every generated record is run through its **zod schema** (`lib/schemas/`) in dev (`schema.parse()`), so fixtures can never drift from the [domain model](05-domain-model.md). Full schemas live in [data model & mock data](06-data-model-and-mock-data.md).

### 4b. Simulated action runner (`mock/runner.ts`)
Turns a dispatched chain into `Run` records with believable latency and weighted outcomes, streaming output for the run console.

```ts
interface RunnerOptions { failureRate?: number; minMs?: number; maxMs?: number; }

export async function runStep(step: ChannelStep, assetId: string, opts: RunnerOptions): Promise<Run> {
  const r = rng(`runner:${assetId}:${step.uid}`);            // deterministic-ish per (asset,step)
  const latency = lerp(opts.minMs ?? 600, opts.maxMs ?? 4000, r());   // simulate work
  await delay(latency);                                      // runtime async — randomness OK here
  const outcome = weighted(r, { succeeded: 1 - (opts.failureRate ?? 0.12), failed: opts.failureRate ?? 0.12 });
  // destructive actions (BMR, isolate, force chain rebuild) get a higher mock failureRate + AlertDialog gate upstream
  return buildRun({ assetId, step, status: outcome, durationSec: latency / 1000, output: mockConsole(step, outcome) });
}
```

- **Chains** run steps sequentially; a failed step can halt the chain or continue per the step config — surfaced as bulk progress ("Running 7 of 40…") via `Progress` + Sonner.
- **Outcomes** are weighted by action category (diagnostics rarely fail; destructive/restore actions fail more) so demos show both success and failure paths.
- **Output** is fabricated console text appropriate to the product (e.g. ZFS pool scrub lines, off-site sync bytes, Graph API throttle notice for SaaS Protect).
- New `Run` records append to the in-memory run feed and the `approvals`/`automation-policies` stores as appropriate; a toast confirms dispatch.

### 4c. Fake "realtime" updates
A single client-side ticker (mounted in the console layout, inside `useEffect`) fabricates life so the console feels live — **runtime randomness only, never during render**:

- A queued/running `Run` advances its status and streams a line every interval until it settles.
- A `Syncing` backup job ticks its progress; an off-site sync completes.
- If an `always` auto-remediation policy exists, the ticker can fabricate a matching alert and **auto-fire** the policy (creating a `Run` with `triggeredBy: 'policy-auto'`), demonstrating standing remediation.
- New critical alerts can drip into the triage queue to exercise the "new" state + badge counts.

The ticker is **seeded-but-runtime**: it uses `rng('ticker')` for reproducible sequencing across a session but lives entirely in effects so it never touches SSR.

## 5. Token & lint ENFORCEMENT (the mandates as machinery)

The mandates ([tokens-only, Storybook-first, impeccable, WCAG 2.2 AA](03-design-system.md)) are enforced by tooling, not goodwill. Concrete rule names and config sketches below.

### 5a. Ban hardcoded color & spacing literals
**Stylelint** guards CSS / `@theme`; **ESLint** guards JSX/TS (Tailwind arbitrary values and inline hex).

`stylelint.config.mjs`:
```js
export default {
  plugins: ['stylelint-declaration-strict-value', 'stylelint-no-unsupported-browser-features'],
  rules: {
    'color-no-hex': true,                         // no raw #hex anywhere
    'color-named': 'never',                       // no `red`, `white`
    // every color/spacing/radius/shadow MUST resolve to a var(--token)
    'scale-unlimited/declaration-strict-value': [
      ['/color/', 'fill', 'stroke', 'background', 'border-color', 'box-shadow',
       'padding', 'margin', 'gap', 'border-radius'],
      { ignoreValues: ['transparent', 'currentColor', 'inherit', 'none', '0'],
        expandShorthand: true,
        message: 'Use a semantic token (var(--…)) — no literals. Tokens are law.' },
    ],
  },
  // EXCEPTION: globals.css is the ONE file allowed to declare raw hex/OKLCH (the palette source).
  overrides: [{ files: ['**/globals.css'], rules: { 'color-no-hex': null,
                'scale-unlimited/declaration-strict-value': null } }],
};
```

`eslint.config.mjs` (flat) — Tailwind + literal bans:
```js
import tailwind from 'eslint-plugin-better-tailwindcss';   // or eslint-plugin-tailwindcss (v4-aware)
import noInlineStyle from 'eslint-plugin-react';
export default [
  {
    plugins: { 'better-tailwindcss': tailwind },
    rules: {
      // ban arbitrary Tailwind values: bg-[#0E67F5], p-[13px], text-[14.5px], gap-[7px]
      'better-tailwindcss/no-unregistered-classes': 'error',
      'better-tailwindcss/enforce-consistent-class-order': 'warn',
      // custom rule: forbid arbitrary color/length brackets in className
      'no-restricted-syntax': ['error', {
        selector: "Literal[value=/\\b(bg|text|border|fill|stroke|ring|from|to|via)-\\[#?[0-9a-fA-F]{3,8}\\]/]",
        message: 'No arbitrary color literal in className — use a semantic token utility (bg-card, text-primary, …).',
      }, {
        selector: "Literal[value=/\\b(p|m|gap|space|w|h|rounded)-\\[[0-9.]+(px|rem)\\]/]",
        message: 'No arbitrary spacing/radius literal — use the scale (p-4, gap-2, rounded-lg).',
      }],
      'react/forbid-dom-props': ['error', { forbid: [{ propName: 'style',
        message: 'No inline style — use token utilities.' }] }],
    },
  },
];
```

Net effect: the **only** place raw `#0E67F5` / OKLCH may appear is `globals.css` (the palette source, stylelint-exempted). Component code references `bg-primary`, `text-status-failed`, `border` (hairline), `rounded-lg` — semantic tokens only. This directly enforces the "tokens are law / no hardcoded values" mandate and the impeccable bans on gradient text / side-stripe borders (no `from-…`/`via-…` literals, no arbitrary `border-l-[…]` colors).

### 5b. CI gate: every component has a story (100% coverage)
A custom Node script in CI asserts each component file has a sibling story. Fails the build on any orphan.

`scripts/check-story-coverage.mjs`:
```js
import { globby } from 'globby';
const components = await globby(['src/components/**/*.tsx',
  '!**/*.stories.tsx', '!**/*.test.tsx', '!**/index.ts']);
const missing = components.filter(f => {
  const story = f.replace(/\.tsx$/, '.stories.tsx');
  return !require('fs').existsSync(story);
});
if (missing.length) {
  console.error('Components missing a *.stories.tsx:\n' + missing.map(m => '  ✗ ' + m).join('\n'));
  process.exit(1);
}
console.log(`✓ Story coverage: ${components.length}/${components.length} components have stories.`);
```

Plus the Storybook quality gates (see [Storybook design system](storybook-design-system.md)): `vitest --project=storybook` (browser mode + Playwright, runs every `play`), `@storybook/addon-a11y` with `a11y.test:'error'` (WCAG 2.2 AA), and Chromatic visual regression.

### 5c. The impeccable hook role
[`impeccable`](storybook-design-system.md) is the design-judgment layer the linters can't encode. Its role in this architecture:

- **Pre-build gate** on any UI work: enforce the absolute bans the rules above can't fully express — no nested cards, no decorative glassmorphism, no hero-metric template, no identical card grids, no per-section uppercase eyebrows, no side-stripe borders, no gradient text.
- **Atomic-design discipline**: every new surface starts Foundations→Atoms→Molecules→Organisms→Templates→Pages with a story per step.
- **Token verification**: a manual/agent check that semantic tokens (not literals) were used and that status is never color-only (dot + icon + label).

Run impeccable on every component/page PR before merge; the stylelint/eslint/coverage gates are the automated floor, impeccable is the ceiling.

### 5d. Accessibility & type gates (rounding out CI)
- `tsc --noEmit` (strict) — model integrity.
- `eslint-plugin-jsx-a11y` recommended set — keyboard/ARIA floor; Storybook a11y addon is the deeper per-component audit.
- WCAG 2.2 AA contrast is *designed in* via OKLCH token pairs (light + slate dark); the a11y addon flags regressions.

## 6. Theming wiring

One source of truth (`globals.css`) flows to Tailwind utilities **and** Storybook, so app and stories render identically.

```

─────────────────                    ────────────────                ─────────────
:root { --primary: <Curious Blue> }  @theme inline {                 .storybook/preview.ts

--status-failed / --warning / …        --color-status-failed: …          withThemeByClassName({

        └──── single declaration ────────────┴──── bg-primary / text-status-failed / fill-chart-1 ───┘

  --chart-1: …; --chart-2: …; --chart-3: …; --chart-4: …; --chart-5: …;
}
.dark { /* slate: subtle teal cast, NOT carbon */ … }

@theme inline {                /* expose every token as a Tailwind utility */
  --color-primary: var(--primary);
  --color-sidebar: var(--sidebar);
  --color-status-failed: var(--status-failed);
  /* …one mapping per token → bg-primary, text-status-failed, bg-sidebar, fill-chart-1 … */

```

- **App**: `app/layout.tsx` wraps `ThemeProvider` (`next-themes`, class strategy, `defaultTheme="dark"` for night-ops feel, `disableTransitionOnChange`); a topbar `DropdownMenu` toggles Light/Dark/System; fonts loaded via `next/font` (Plus Jakarta Sans display, Figtree body) and exposed as token-mapped font utilities.
- **Storybook**: `.storybook/preview.ts` imports the same `globals.css` and uses `@storybook/addon-themes` `withThemeByClassName({ themes:{ light:'', dark:'dark' } })` — the `.dark` class mechanism is shared, so no token is ever duplicated.
- **Status→token map** (`lib/status.ts`) is the single mapping from `AssetStatus`/`Severity` → `{ token, icon, label }`, guaranteeing status is never color-only and severity sort is consistent everywhere.

## 8. Performance — large mock fleets

Hundreds-to-thousands of rows across fleet/backups/alerts/runs. Keep it buttery:

- **Row virtualization** with `@tanstack/react-virtual` on every primary table; render only visible rows + overscan. Sticky header/col remain pinned (the dense table is the core surface).
- **Client-side filter/sort/paginate** in `mock/query.ts` returns only the page slice — the grid never receives the whole fixture array.
- **Memoize** column defs (module-level, never inline) and derived facet counts; `select`-style Zustand subscriptions so a param edit in the cart doesn't re-render the table.
- **Stable keys** from deterministic IDs (`AST-000142`) — no index keys, so virtualization + reorder stay correct.
- **Defer heavy surfaces**: charts and the run console are `next/dynamic`-imported where below the fold; the "last 10 backups" dot-strip is a cheap SVG/`<span>` row, not a chart.
- **Debounce** search input before pushing to nuqs (avoid a URL write per keystroke).
- **Hydration safety**: persisted cart-count badge renders only after `useHasHydrated()` to avoid mismatch and an extra reconcile.
